{{% alert note %}}
**DEF.** An **interrupt** is an automatic transfer of software execution.
{{% /alert %}}

In particular, {{< hl >}}in response to an hardware event one has an hardware trigger which cause the execution of a software response.{{< /hl >}} This software response is called _Interrupt Service Routine_ (**ISR**).

This process is _asynchronous_ with the current software execution. Also, interrupts are the base of _multi-threaded programming,_ in which one can have various threads without having a super complicated unique program.

They can be used with (1) external I/Os, (2) internal events (e.g. memory fault) and (3) periodic events (e.g. timers).

Even though the usage of global variables is not recommended, in this case they are used to easily share informations between threads.
{{% /alert %}}

## Interrupt Processing

1. HW trigger.
- The current instruction is finished.
- The execution of the main threas is suspended and 8 registers are pushed on the stack (PSR, PC, LR, R12, R3, R2, R1, R0).
- The link register (LR) is set to $0xFFFFFFF9$. It indicates that an interrupt is running.
- The IPSR register is set to the interrupt number (depending on the kind of interrupt).
- PC is set to ISR address and the interrupt is started.
- The flag that requested the interrupt is cleared.
- The code written for the interrupt is performed.
- Global variables are red and set.
- The BXLR ("branch link") instruction is executed so that 8 registers are pulled from the stack and the main thread is resumed.

Each kind of interrupt is defined in a file of the SDK. In the case of the MSP432, the file is called `startup_msp432p401r_ccs.c`. {{< hl >}}With each type of interrupt, also a function name is provided. When this function is defined in the codebase, it will be executed every time that interrupt is called.{{< /hl >}}

---
## Interrupt Priority

One can define the interrupt importance with a _3-bit number: 0 is high, 7 is low._ This will determine wether, if while running an interrpt another interrupt is triggered, that will be executed or postponed.

---
## Periodic Interrupts

{{< hl >}}Periodic interrupts can be generated by timers.{{< /hl >}}

On the MSP432, for example, it can be generated by the 1&rarr;0 transition of the `VAL` counter of the SysTick timer. To activate it, the bit 1 of the `CTRL` register (called `INTEN`) should be set to 1.

## Critical Sections

{{% alert warning %}}
Whenever multiple threads (1) share permanent variables/permissions (e.g. port access,...) and (2) can write/change those informations, a **critical section** is encountered.
{{% /alert %}}

In fact, as explained in lesson 10.3 of the TI-RSLK Max course (see References), it is possible that, if an interrupt is executed in the middle of another one (and both access the same registers), both of them will be completed but the result is different from what expected.

This happens because both uses the same registers to access the same informations, but those registers are not cleared and restored to original after the second interrupt is completed!.

There are vaious possible solution to this inconvenience:

1. don't access the same port/variable with two different interrupts;
- disable other interrupts &rarr; access informations &rarr; re-enable other interrupts;
- **make them the same priority** so they will execute one after the other in any circumstance.

{{% alert note %}}
**DEF.** A code sequence is defined **atomic sequence** if it is executed in one go, withouth possibility of interruption.

_If an interrupt is non-atomic, critical sections can happen._ 
{{% /alert %}}
